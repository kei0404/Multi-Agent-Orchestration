あなたはエンジニアです。以下のひな型をベースにしてClaude Code の実行ルールとなる `CLAUDE.md` ファイルを作成してください。作成にあたり、以下の２つの原則を参照し、５つのコツに従って作成してください。

## 2つの原則/5つのコツ
### 2つの原則
- LLMはステートレスである: 
  - プロジェクトごとに**オンボーディング（導入研修）を行う**
  - AIに対してプロジェクトのルールを教え、即座に作業可能な状態にするための**オンボーディング資料**としての役割を与える
- コンテキストウィンドウは有限のリソースである: 
  - Aユー０ザテキストウィンドウ）を設定や指示で埋めすぎない
  - **コンパクション（圧縮）**の回避のため、設定は最小限にし、思考のためのスペースを最大限残す

### 5つのコツ
- コツ1: Less is More（短く保つ） 
  - CLAUDE.mdは短く保つ
  - 目安として300語程度に抑えることが推奨
- コツ2: Progressive Disclosure（段階的開示） 
  - 必要な時に必要な情報を参照させる
  - 起動時にすべてを読み込ませるのではなく、本当に必要な情報以外は書かない
  - 特定のツールの詳細な使い方はskillsやrulesといった別ファイルに逃がす
- コツ3:  What, Why, Howを書く（特にHowを厚く）
  -  「何のためのプロジェクトか（What/Why）」よりも、AIに**「どう振る舞ってほしいか（How）」**を重点的に記述する。
  - 回答スタイル（前置きを省く等）やツールの優先順位などを具体的に指定する
  - README.mdとの重複は避ける
- コツ4: 検証手段（完了条件）を明記する
  - テストの実行、型チェック、リンターの実行など、**どうすればタスク完了と言えるのかという「検証手段」**を記述する。
- コツ5:  運用して削る
  - 最初から完璧を目指さず、実際の失敗や不便さに基づいて内容を更新する
  - ユーザーが運用しながら不要な記述を削り、洗練させていけるように必要な情報は網羅する

## 出力形式
- docs/templates 内のテンプレートを参照させる
- テンプレートは `.claude/docs/templates/claude-orchestration.md` を使用する。

## CLAUDE.mdを作成する際に考慮する点
- 指示はできるだけ少なくする
- 汎用的な指示/ルールを心がける
- 指示が長くなりすぎたら、別のファイルに分ける（**段階的開示**）
- 日本語で記述する

## いつPLANSを参照するかの規定
- PLANS.md が唯一の決定事項の所在とする
- `./docs/plans` 内の `PLANS.md` を常に参照する

## ファイルの保存場所
プロジェクトルートに保存する

## ルール
- `CLAUDE.md` 作成にあたって、不明点があればユーザーに質問する
- `spec.md` が記述されていれば参照する
