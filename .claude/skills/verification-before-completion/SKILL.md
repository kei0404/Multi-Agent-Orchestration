# 完了前の検証

## 概要

検証なしに「作業完了」と主張することは効率ではなく誠実さの欠如である。

**核心原則：** 主張より先に証拠、常に。

**このルールの文字に反することは、その精神にも反することである。**

## 鉄則

```
新しい検証結果がなければ、完了の主張は許されない
```

このメッセージ内で検証コマンドを実行していなければ、「合格」とは主張できない。

## ゲート機能

```
いかなる状態や満足を主張する前に：

1. IDENTIFY（特定）：この主張を裏付けるコマンドは何か？
2. RUN（実行）：フルのコマンドを実行する（新規・完全に）
3. READ（確認）：出力全文を読み、終了コードと失敗件数を確認する
4. VERIFY（検証）：出力は主張を裏付けているか？
   - NO の場合：証拠とともに実際の状態を報告する
   - YES の場合：証拠とともに主張を述べる
5. その後で初めて：主張を行う

どのステップをスキップしても、それは検証ではなく虚偽である
```

## よくある失敗

| 主張 | 必要とするもの | 不十分なもの |
|------|----------------|--------------|
| テストが通る | テスト実行結果：失敗 0 件 | 以前の実行、「通るはず」 |
| リントがクリーン | リント出力：エラー 0 件 | 部分的チェック、推測 |
| ビルドが成功 | ビルドコマンド：終了コード 0 | リント通っている、ログが良さそう |
| バグが修正済み | 元の症状のテスト：合格 | コードを変更した、直ったはず |
| リグレッションが有効 | Red-Green サイクルを確認済み | 一度テストが通った |
| エージェント完了 | VCS の diff で変更を確認 | エージェントの「成功」報告 |
| 要件を満たした | 行ごとのチェックリスト | テストが通っている |

## 警告サイン - 停止

- 「should」「probably」「seems to」の使用
- 検証前に満足を表明する（「素晴らしい！」「完璧！」「完了！」など）
- 検証なしにコミット／プッシュ／PRしようとしている
- エージェントの成功報告を鵜呑みにする
- 部分的な検証に頼る
- 「今回だけ」と思い込む
- 疲れて早く終わらせたい
- **検証を実行せずに成功を示唆する表現はすべて禁止**

## 合理化の防止

| 言い訳 | 現実 |
|--------|------|
| 「今度は動くはず」 | 検証コマンドを実行する |
| 「自信がある」 | 自信 ≠ 証拠 |
| 「今回だけ」 | 例外は認めない |
| 「リントは通った」 | リント ≠ コンパイラ |
| 「エージェントが成功と言った」 | 自分で検証する |
| 「疲れている」 | 疲労は言い訳にならない |
| 「部分的チェックで十分」 | 部分は何も証明しない |
| 「表現が違うからルールは適用されない」 | 精神が文字に優先する |

## 重要なパターン

**テスト：**
```
✅ [テストコマンド実行] [34/34 成功を確認] 「全テスト合格」
❌ 「今度は通るはず」「正しそうに見える」
```

**リグレッションテスト（TDD Red-Green）：**
```
✅ 作成 → 実行（合格） → 修正を戻す → 実行（必ず失敗） → 復元 → 実行（合格）
❌ 「リグレッションテストを作成した」（Red-Green の検証なし）
```

**ビルド：**
```
✅ [ビルド実行] [終了コード 0 を確認] 「ビルド合格」
❌ 「リントは通った」（リントはコンパイルをチェックしない）
```

**要件：**
```
✅ 計画を再読 → チェックリスト作成 → 各項目を検証 → ギャップまたは完了を報告
❌ 「テスト通ったから、フェーズ完了」
```

**エージェントへの委譲：**
```
✅ エージェントが成功報告 → VCS diff を確認 → 変更を検証 → 実際の状態を報告
❌ エージェントの報告を信じる
```

## なぜ重要か

24件の失敗事例から：
- 人間のパートナーが「信じられない」と言った - 信頼が損なわれた
- 未定義の関数がリリースされた - クラッシュする
- 欠けている要件のままリリースされた - 不完全な機能
- 誤った完了宣言のための時間浪費 → 方向転換 → 手戻り
- 「誠実さは核心的価値である。嘘をつけば、交代される」に違反する

## 適用タイミング

**次の前に必ず：**
- 成功・完了を示すいかなる表現
- いかなる満足の表明
- 作業状態に関するいかなる肯定的な発言
- コミット、PR作成、タスク完了の宣言
- 次のタスクへの移行
- エージェントへの委譲

**ルールが適用されるのは：**
- そのままの文言
- 言い換えや同義表現
- 成功の含み
- 完了・正しさを示唆するいかなる発言

## 結論

**検証にショートカットは許されない。**

コマンドを実行する。出力を読む。その上で結果を主張する。

これは譲れない。
