# 実装計画の作成

## 概要

コードベースのコンテキストがゼロのエンジニアを想定した包括的な実装計画を作成する。各タスクで触れるファイル、コード、テスト、参照すべきドキュメント、テスト方法など、必要な情報をすべてドキュメント化する。計画は一口サイズのタスクとして提示する。DRY・YAGNI・TDD・頻繁なコミットを原則とする。

担当エンジニアは熟練した開発者だが、ツールセットや問題領域についてはほとんど知らないと想定する。また、テスト設計についてはあまり詳しくないと想定する。

**開始時のアナウンス:** 「writing-plansスキルを使用して実装計画を作成します。」

**コンテキスト:** このスキルは専用のworktree（brainstormingスキルで作成）で実行することを推奨。

**保存先:** `docs/plans/PLANS.md`

## 作業手順

1. `docs/specs/spec.md` を読み込む
2. リポジトリルートを確認する
3. `docs/plans/PLANS.md` が存在しない場合は新規作成し、存在する場合はテンプレートに沿って差分をマージする
4. 技術スタックを確認する
5. 機能をタスクに分解する
6. 依存関係を整理する
7. `docs/plans/PLANS.md` として保存する
8. 先頭数行をプレビューとしてユーザーに提示し、完了を報告する

## 4つの指導原則

**原則1 自己完結型:** 相手に期待せず、前提知識ゼロでも理解できる状態にする。

**原則2 自己充足型:** 進捗に応じて書き換えるが、自己完結を常に維持する。

**原則3 初心者誘導型:** 初心者に手取り足取り教えるように開発のレールを敷く。

**原則4 成果重視型:** 動作することが最優先。目に見える結果で判断する。

## タスク分解ルール

- 1タスク = 1〜2時間（各ステップは1アクション 2〜5分）
- チェックボックス形式（`[ ]`）
- 完了条件を必ず記載

## PLANS.md 構成

PLANS.md は以下の15セクションを必ず含む。テンプレート: `.claude/docs/templates/plans.md`

```markdown
# [プロジェクト名] PLANS.md

## 1. メタデータブロック
- プロジェクト名 / バージョン / ライセンス

## 2. PLANS.md の使用方法
- ExecPlan作成時はPLANS.mdを忠実に守る
- 実装時はユーザーに「次のステップ」を尋ねず次のマイルストーンへ進む
- 決定事項は必ずログに記録する
- 困難な要件には概念実証を実施する

## 3. 目的 / 背景 / 全体像
- プロジェクト目標 / 価値提案 / 解決する問題

## 4. 文脈と方向性
- 問題の背景 / 設計理念 / 主な制約 / キーファイル（フルパス）

## 5. 仕様
- システム仕様 / アーキテクチャ決定 / コア機能仕様

## 6. 目標とする成果物
- spec.md を参照した成果物の定義

## 7. 進捗
- [ ] チェックボックス付きタスク（完了/残存をタイムスタンプ付きで記録）

## 8. 驚きと発見
- 予期せぬ挙動・バグ・洞察（証拠スニペット付き）

## 9. 決定ログ
- 日付付き決定事項 / 根拠 / トレードオフ

## 10. 作業計画
- 編集と追加の順序（ファイル名・関数・モジュール・変更内容を具体的に）

## 11. 具体的なステップ
- 実行コマンド / 作業ディレクトリ / 期待される出力

## 12. 未解決の質問
- 未解決事項 / 検討中の選択肢 / 阻害要因

## 13. 成果と振り返り
- 達成されたこと / 課題 / 教訓

## 14. 検証と受け入れ
- 起動方法 / 観察すべき動作 / 受け入れ基準 / テストシナリオ / 成功指標
- 記入例：「テストコマンドを実行し、新しいテストが合格する」

## 15. 更新履歴
```

## タスク構造

````markdown
### タスクN: [コンポーネント名]

**ファイル:**
- 作成: `exact/path/to/file.py`
- 変更: `exact/path/to/existing.py:123-145`
- テスト: `tests/exact/path/to/test.py`

**ステップ1: 失敗するテストを書く**

```python
def test_specific_behavior():
    result = function(input)
    assert result == expected
```

**ステップ2: テストを実行して失敗を確認**

実行: `pytest tests/path/test.py::test_name -v`
期待結果: FAIL（"function not defined"）

**ステップ3: 最小限の実装を書く**

```python
def function(input):
    return expected
```

**ステップ4: テストを実行してパスを確認**

実行: `pytest tests/path/test.py::test_name -v`
期待結果: PASS

**ステップ5: コミット**

```bash
git add tests/path/test.py src/path/file.py
git commit -m "feat: add specific feature"
```
````

## 作成上の注意事項

- PLANS.md 本文は 500〜600 行程度に収める
- 600行を超える場合は内容別に別 `.md` ファイルを作成し参照する（ユーザー承認が必要）
- 関連ファイルは `./docs/plan-files/` に保存する
- 常に正確なファイルパスを記載
- 計画には完全なコードを含める（曖昧な表現は避ける）
- 正確なコマンドと期待される出力を記載
- DRY、YAGNI、TDD、頻繁なコミット

## 実行の引き継ぎ

計画を保存した後、実行方法の選択を提示:

**「計画が完了し、`docs/plans/PLANS.md` に保存しました。2つの実行オプションがあります:**

**1. サブエージェント駆動（このセッション）** - タスクごとに新しいサブエージェントを起動、タスク間でレビュー、高速な反復

**2. 並列セッション（別セッション）** - executing-plansで新しいセッションを開き、チェックポイント付きのバッチ実行

**どちらのアプローチを選びますか？」**

**サブエージェント駆動を選択した場合:**
- **必須サブスキル:** superpowers:subagent-driven-developmentを使用
- このセッションに留まる
- タスクごとに新しいサブエージェント + コードレビュー

**並列セッションを選択した場合:**
- worktreeで新しいセッションを開くよう案内
- **必須サブスキル:** 新しいセッションでsuperpowers:executing-plansを使用
