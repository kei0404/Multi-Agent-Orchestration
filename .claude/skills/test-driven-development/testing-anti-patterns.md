# テストのアンチパターン

**このリファレンスを読むタイミング：** テストの作成・変更時、モック追加時、またはプロダクションコードにテスト専用メソッドを追加したくなったとき。

## 概要

テストは実際の動作を検証すべきであり、モックの動作ではない。モックは分離のための手段であり、テスト対象そのものではない。

**核心原則：** コードが何をするかをテストせよ。モックが何をするかではなく。

**厳密なTDDに従えばこれらのアンチパターンは防げる。**

## 鉄則

```
1. モックの動作をテストしない
2. プロダクションクラスにテスト専用メソッドを追加しない
3. 依存関係を理解せずにモックしない
```

## アンチパターン1: モックの動作をテストしている

**違反例：**
```typescript
// ❌ 悪い例: モックの存在をテストしている
test('renders sidebar', () => {
  render(<Page />);
  expect(screen.getByTestId('sidebar-mock')).toBeInTheDocument();
});
```

**なぜ間違いか：**
- コンポーネントではなくモックが動作するかを検証している
- モックがあればテストが通り、なければ失敗する
- 実際の動作について何もわからない

**人間のパートナーの指摘：**「モックの動作をテストしていないか？」

**修正例：**
```typescript
// ✅ 良い例: 実際のコンポーネントをテストするか、モックしない
test('renders sidebar', () => {
  render(<Page />);  // サイドバーをモックしない
  expect(screen.getByRole('navigation')).toBeInTheDocument();
});

// もしくは分離のためにサイドバーをモックする必要がある場合：
// モックに対してアサートしない - サイドバーがある状態でのPageの動作をテストする
```

### ゲート機能

```
モック要素に対してアサートする前に：
  問う：「実際のコンポーネントの動作をテストしているか、それともモックの存在だけか？」

  モックの存在をテストしている場合：
    停止 - アサーションを削除するか、モックを外す

  代わりに実際の動作をテストする
```

## アンチパターン2: プロダクションコードにテスト専用メソッド

**違反例：**
```typescript
// ❌ 悪い例: destroy() はテストでしか使われない
class Session {
  async destroy() {  // プロダクションAPIに見える！
    await this._workspaceManager?.destroyWorkspace(this.id);
    // ... クリーンアップ
  }
}

// テスト内
afterEach(() => session.destroy());
```

**なぜ間違いか：**
- プロダクションクラスがテスト専用コードで汚染される
- プロダクションで誤って呼び出されると危険
- YAGNIと関心の分離に違反
- オブジェクトのライフサイクルとエンティティのライフサイクルを混同

**修正例：**
```typescript
// ✅ 良い例: テストユーティリティがテストのクリーンアップを担当
// Session に destroy() はない - プロダクションではステートレス

// test-utils/ 内
export async function cleanupSession(session: Session) {
  const workspace = session.getWorkspaceInfo();
  if (workspace) {
    await workspaceManager.destroyWorkspace(workspace.id);
  }
}

// テスト内
afterEach(() => cleanupSession(session));
```

### ゲート機能

```
プロダクションクラスにメソッドを追加する前に：
  問う：「テストでしか使わないのではないか？」

  はいの場合：
    停止 - 追加しない
    代わりにテストユーティリティに置く

  問う：「このクラスがこのリソースのライフサイクルを所有しているか？」

  いいえの場合：
    停止 - このメソッドを置くクラスが間違っている
```

## アンチパターン3: 理解なきモック

**違反例：**
```typescript
// ❌ 悪い例: モックがテストのロジックを壊している
test('detects duplicate server', () => {
  // テストが依存する設定書き込みをモックが阻止している！
  vi.mock('ToolCatalog', () => ({
    discoverAndCacheTools: vi.fn().mockResolvedValue(undefined)
  }));

  await addServer(config);
  await addServer(config);  // 例外が投げられるべき - だが投げられない！
});
```

**なぜ間違いか：**
- モックしたメソッドにテストが依存する副作用（設定書き込み）があった
- 「念のため」の過剰なモックが実際の動作を壊す
- テストが間違った理由で通るか、不可解な失敗をする

**修正例：**
```typescript
// ✅ 良い例: 適切なレベルでモック
test('detects duplicate server', () => {
  // 遅い部分だけモックし、テストに必要な動作は保持
  vi.mock('MCPServerManager'); // 遅いサーバー起動だけモック

  await addServer(config);  // 設定が書き込まれる
  await addServer(config);  // 重複が検出される ✓
});
```

### ゲート機能

```
メソッドをモックする前に：
  停止 - まだモックしない

  1. 問う：「実際のメソッドにはどんな副作用があるか？」
  2. 問う：「このテストはそれらの副作用に依存しているか？」
  3. 問う：「このテストが何を必要としているか完全に理解しているか？」

  副作用に依存している場合：
    より低レベルでモックする（実際に遅い/外部の操作を）
    または必要な動作を保持するテストダブルを使う
    テストが依存する高レベルメソッドをモックしない

  テストが何に依存しているか不明な場合：
    まず実際の実装でテストを実行する
    何が実際に起こる必要があるか観察する
    その後、適切なレベルで最小限のモックを追加する

  警告サイン：
    - 「念のためモックしておこう」
    - 「遅いかもしれないからモックしよう」
    - 依存関係チェーンを理解せずにモックしている
```

## アンチパターン4: 不完全なモック

**違反例：**
```typescript
// ❌ 悪い例: 部分的なモック - 必要と思うフィールドだけ
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' }
  // 欠落: 下流のコードが使用するmetadata
};

// 後で: コードが response.metadata.requestId にアクセスして壊れる
```

**なぜ間違いか：**
- **部分的なモックは構造的な仮定を隠す** - 知っているフィールドだけモックした
- **下流のコードが含めなかったフィールドに依存する可能性** - サイレントな失敗
- **テストは通るが統合で失敗** - モックは不完全、実際のAPIは完全
- **偽の安心感** - テストが実際の動作について何も証明しない

**鉄則：** テストが直接使うフィールドだけでなく、現実に存在する完全なデータ構造をモックする。

**修正例：**
```typescript
// ✅ 良い例: 実際のAPIの完全性を再現
const mockResponse = {
  status: 'success',
  data: { userId: '123', name: 'Alice' },
  metadata: { requestId: 'req-789', timestamp: 1234567890 }
  // 実際のAPIが返すすべてのフィールド
};
```

### ゲート機能

```
モックレスポンスを作成する前に：
  確認：「実際のAPIレスポンスにはどんなフィールドがあるか？」

  アクション：
    1. ドキュメント/サンプルから実際のAPIレスポンスを確認する
    2. システムが下流で使用する可能性のあるすべてのフィールドを含める
    3. モックが実際のレスポンススキーマと完全に一致するか検証する

  重要：
    モックを作成するなら、構造全体を理解する必要がある
    部分的なモックは、省略されたフィールドにコードが依存するとき静かに失敗する

  不確かなら：ドキュメントに記載されたすべてのフィールドを含める
```

## アンチパターン5: 後付けの結合テスト

**違反例：**
```
✅ 実装完了
❌ テスト未作成
「テストの準備完了」
```

**なぜ間違いか：**
- テストは実装の一部であり、オプションのフォローアップではない
- TDDならこれを防げた
- テストなしに完了とは言えない

**修正例：**
```
TDDサイクル：
1. 失敗するテストを書く
2. 通るように実装する
3. リファクタリングする
4. その後で完了と宣言する
```

## モックが複雑になりすぎたとき

**警告サイン：**
- モックのセットアップがテストロジックより長い
- テストを通すためにすべてをモックしている
- モックに実際のコンポーネントが持つメソッドが欠けている
- モックを変更するとテストが壊れる

**人間のパートナーの問い：**「ここでモックを使う必要があるのか？」

**検討：** 複雑なモックより、実際のコンポーネントを使った結合テストのほうがシンプルなことが多い

## TDDがこれらのアンチパターンを防ぐ理由

**TDDが役立つ理由：**
1. **テストを先に書く** → 実際に何をテストしているか考えることを強制する
2. **失敗を確認する** → テストがモックではなく実際の動作をテストしていることを確認
3. **最小限の実装** → テスト専用メソッドが忍び込まない
4. **実際の依存関係** → モックする前にテストが実際に何を必要としているか見える

**モックの動作をテストしているなら、TDDに違反している** - 実際のコードに対してテストが失敗するのを確認する前にモックを追加した。

## クイックリファレンス

| アンチパターン | 修正方法 |
|----------------|----------|
| モック要素に対してアサート | 実際のコンポーネントをテストするかモックを外す |
| プロダクションにテスト専用メソッド | テストユーティリティに移動 |
| 理解なきモック | まず依存関係を理解し、最小限にモック |
| 不完全なモック | 実際のAPIを完全に再現 |
| 後付けのテスト | TDD - テストファースト |
| 過度に複雑なモック | 結合テストを検討 |

## 警告サイン

- `*-mock` のテストIDに対するアサーション
- テストファイルからしか呼ばれないメソッド
- テストの50%以上がモックセットアップ
- モックを削除するとテストが失敗する
- なぜモックが必要か説明できない
- 「念のため」モックしている

## 結論

**モックは分離のためのツールであり、テスト対象ではない。**

TDDでモックの動作をテストしていると気づいたら、間違った方向に進んでいる。

修正方法：実際の動作をテストするか、そもそもなぜモックしているか問い直す。
