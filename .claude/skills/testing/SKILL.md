# テストのベストプラクティス

## テスト構造（AAAパターン）
- Arrange: テストデータと前提条件を準備する
- Act: テスト対象のコードを実行する
- Assert: 期待される結果を検証する
- 説明的なテスト名を付ける（例: \"should X when Y\"）

## 単体テスト（Unit Tests）
- 1つのテストにつき、1つの振る舞いだけを検証する
- 外部依存（API・DB・外部サービスなど）はモックする
- エッジケースとエラーパスもテストする
- 各テストは高速に保つ（目安: 100ms 未満）
- テストデータにはファクトリを活用する
- カバレッジは 80% 以上を目標とする

## 結合テスト（Integration Tests）
- 複数コンポーネント／モジュールの連携を検証する
- テスト用データベースやコンテナを利用する
- テスト後は状態をクリーンアップする
- 正常系（ハッピーパス）だけでなく異常系もテストする
- できるだけ現実に近いデータを使う

## E2E テスト
- クリティカルなユーザーフローに絞ってテストする
- テスト同士が互いに依存しないようにする
- 安定したセレクタ（`data-testid` など）を使用する
- 非同期処理を正しく扱う（待機条件を明示する）
- CI 上で実行し、必要に応じてリトライを設定する

## モックの方針（Mocking）
- モックは境界（API・DB など）で行う
- テストデータ生成にはファクトリを使う
- モックしすぎない（実際の振る舞いを検証できる範囲を残す）
- テスト間でモック状態を必ずリセットする
- 可能であればモックよりもフェイク（簡易実装）を優先する

## ツール
- Jest / Vitest: 単体テスト
- Playwright / Cypress: E2E テスト
- MSW: API モック
- Faker: テストデータ生成
- Testing Library: コンポーネントテスト
